## 1. Where Accounting actually is (the honest truth)

The owner believes Accounting is near done. It is not — but the reason is encouraging: **the hard part is built, the visible part is not.**

| Layer | Reality | Evidence |
|---|---|---|
| **Database schema** | ~90% complete | `0002_accounting_schema.sql`: currencies, account_types, accounts (composite-FK tenant safety), accounting_periods (gist no-overlap), journal_entries/lines (dual-currency CHECKs), customers/suppliers, invoices/bills + lines, banking, the `general_ledger` view filtered to `status='posted'`. |
| **Posting engine** | Done & rigorous | `0004_functions_posting.sql`: `post_journal_entry()`, `reverse_journal_entry()`, `next_number()`, `fx_rate()`, immutability triggers on posted rows, `audit_trigger` on financial tables. |
| **Security (RLS)** | Done | `0003_rls_and_helpers.sql` gates every table by `core.has_permission(company_id, key)`; seed has the full permission catalogue + 5 system roles. |
| **App data-access layer** | **0% — does not exist** | `src/modules/accounting/` is absent (only `src/modules/cargo-assurance/` exists). No queries, no server actions. |
| **UI screens** | **~3%** | `app/accounting/` is two files: a `ModuleShell` layout and a placeholder `page.tsx` whose two links (`/accounting/accounts`, `/accounting/periods`) **404**. The manifest declares 13 nav sections; **none of the routes exist.** |

**Plain English:** Accounting is a powerful, correct engine with no cockpit. You cannot yet create an account, post a journal, or see a trial balance from the UI.

**Engine gaps the UI build must close** (spec'd in docs, absent in DB — must live in app code/actions):
1. No `set_period_status` RPC — the open→closed→locked state machine must be enforced in a server action over a direct `UPDATE` (and `period.reopen` permission key is **not** seeded — decide: add it, or gate reopen on `periods.manage`).
2. No period-generation routine — fiscal-year period creation done in app code.
3. No starter-chart seed — ship a T&T-oriented default chart in code.
4. No invoice/bill posting RPC — the action layer assembles the balanced draft JE (engine §10.1–10.5), inserts entry+lines, sets `journal_entry_id`, then calls `post_journal_entry`.
5. No company-level control-account / FX-rounding / opening-equity settings store — needs a small `company_settings` table (new migration) or core jsonb.

**Shortest credible path to "close to finished":**
> **(A)** Build `src/modules/accounting/` (data + 4 server actions) once. **(B)** Ship the canonical spine end-to-end — **Setup → Chart of Accounts → Journal Entry (create + post) → Trial Balance** — with the design system applied. That single thread *proves the engine works* and makes the module demonstrably real. **(C)** Fan out to Periods, AR/Invoices, AP/Bills, Banking, Reports, Dashboard. Every screen after the spine is thin over an existing table/view; the engine already guarantees integrity.

## 3. Design direction

Per `docs/DESIGN.md` — **register: Product, mood: trustworthy / precise / calm.** A senior accountant lives in this all day on desktop and checks it on a phone at a fuel terminal.

**Foundational moves (already mostly in tokens — the job is to *consume* them):**
- **Restrained teal.** Brand teal `#0f766e` (`--primary` ramp, OKLCH) as the single accent, ≤~10% of any surface. Neutrals tinted a hair cool toward the brand hue — never the warm AI cream/sand default.
- **Inter, self-hosted, tabular figures.** One family, multiple weights, no pairing. `font-variant-numeric: tabular-nums` on every money/quantity column. **Blocker:** fix the `--font-sans` collision (Milestone 0.1) or none of this renders.
- **Contrast non-negotiable.** Body text on the dark-ink ramp (≥4.5:1); muted is for de-emphasized labels only.
- **Finance semantics by ink weight, not fills.** Debit/credit and positive/negative variance lean on ink weight + `success`/`warning`/`danger`, not red backgrounds.

**Shell (header + module sidebar):**
- **Desktop:** sticky global header (brand mark · company switcher · user cluster) at exactly `--header-h:56px`; left `.module-shell` rail (248px, grouped, sticky, `aria-current` active state).
- **Mobile:** header wraps; sidebar collapses to the existing `.module-shell` **horizontal-scrolling pill nav** above content (no fragile JS drawer) with active-pill `scrollIntoView`. Thumb-first.

**Module screens:**
- **DataTable is the backbone** — sticky header, right-aligned tabular numerics, zebra-free quiet rows, horizontal scroll on mobile (never reflow money into unreadable stacks), optional sticky-left label column, edge scroll-shadow.
- **Forms** single-column on mobile (16px inputs to kill iOS zoom, numeric inputmode on money/qty); aligned multi-column grids on desktop (journal lines: account | debit | credit).
- **Cards used sparingly** — launcher + summary tiles only; never nested, never an endless identical grid. Give launcher cards a glyph + quick facts and cap columns at 2–3.
- **Honest empty states** with a small monochrome glyph in a teal-weak circle. No demo data, ever — a review with no approved loadouts says so per panel.
- **Motion:** Emil curves only (`--ease-out`, `--ease-drawer`), <300ms, press `scale(0.97)`, enter from `scale(0.97)+opacity` (never `scale(0)`), full `prefers-reduced-motion` fallback.

**Anti-slop guardrails (enforced):** no side-stripe accent borders, no gradient text, no decorative glassmorphism beyond the header blur, no hero-metric template, no per-section uppercase eyebrows scattered around, no identical-card grids, no text overflow at any breakpoint. *If it could be flagged "AI made that," rework it.*

## 4. Security guardrails (bake in as modules grow)

The DB is the strong source of truth (RLS via `core.user_companies()`/`core.has_permission()`); the gap is the **entire app-layer authz tier.** Before the nine planned modules and external users arrive:

1. **Server-side route authorization is non-negotiable (P0, 0.4).** `requireModuleAccess(key)` atop every module layout, returning `notFound()` (404, not 403 — don't confirm existence) when the module isn't enabled or the min-view permission is missing. RLS stays authoritative; this is defence-in-depth + correct UX.
2. **Audit the privilege surface (P0, 0.5).** Attach `audit_trigger` to `roles, role_permissions, users, company_modules, clients, documents` and fix the `audit_logs.company_id ON DELETE SET NULL` orphaning so a deleted company can't hide its trail.
3. **Service-role is server-only + centralized (P1, 4.7).** `import 'server-only'`, ESLint allowlist, explicit acting-user id on every privileged mutation, CI guard against `NEXT_PUBLIC_*` secrets.
4. **One portal pattern for all external users (P1, 4.8).** Build `app/(portal)/` now so external client viewers (who currently resolve to `no_company` and get locked out) have a CLIENT-scoped reading surface every future portal inherits.
5. **Fail closed (P1, 4.9).** `getPlatformContext` must distinguish unconfigured / transient-error / auth-error and treat `error` as DENY in guards — never silently degrade to "unconfigured." Generic sign-in errors + throttling to stop enumeration.
6. **Prove isolation in CI (P2, 4.12).** RLS-coverage assertions, no cross-module-schema FKs, a pgTAP suite proving company A can't read/write company B and an external viewer sees only their published snapshot.
7. **Encode lockout invariants (P3, 4.10).** No removing the last super-admin / last `users.manage` holder; no self-escalation; validate module settings against the manifest schema before code trusts them.

## 5. Performance guardrails (bake in as modules grow)

Most perf is about establishing patterns **before** the heavy screens exist:

1. **One network auth per request (P0, 0.2)** and **parallel context reads (P0, 0.3)** — every navigation awaits `getPlatformContext`; these two fixes speed up 100% of page loads and are the biggest mobile wins.
2. **Fix the font collision (P0/P1, 0.1)** — render-blocking 3rd-party origin removed, CLS cut, and tabular money columns finally render.
3. **Keyset pagination, never OFFSET; filter in SQL (P0, 4.1)** on the `general_ledger` 4-table view, which will return tens of thousands of rows.
4. **No N+1 — nested PostgREST selects (P0, 4.2)** as the documented data-access rule; FK indexes already exist.
5. **Aggregate in Postgres (P1, 2.6/1.6)** — dashboards and statements return ~10 summary rows, not raw ledgers; design dashboards to read one aggregate query so a materialized-balances source can swap in later.
6. **Virtualize >100-row lists (P1, 4.3)**, **lazy-load heavy widgets while keeping manifests metadata-only (P1, 4.4)**, **tier caching** (request-scoped + tagged per-company + aggressive for global catalogues) **(P1, 4.5)**, and **stream heavy dashboards behind Suspense skeletons (P2, 4.6)**.
7. **Optimistic posting (P1, 1.5)** with `useOptimistic`/`useFormStatus` so high-latency mobile writes feel instant — presentation only; the posting engine + RLS remain authoritative.
8. **Bound RLS cost (P2, 4.11)** — keep the scalar-subselect pattern; `EXPLAIN ANALYZE` once data exists.

## 6. Logo design brief — TEAL Enterprise

**TEAL = Taylor Engineering Agencies Limited** — a Trinidad & Tobago maritime / engineering / accounting group. Brand teal `#0f766e`. The mark must feel **trustworthy, precise, enterprise** and work both as a small app icon (the `.brand-mark` teal rounded-square already referenced in the shell header / favicon) and as a wordmark.
